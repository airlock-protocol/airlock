India Launched Agentic Payments at National Scale. Who Verifies the Agents?

Shivdeep Singh
Builder — Airlock Protocol

India runs the world's largest real-time payments network. In 2024, UPI processed over 170 billion transactions. This year, AI agents entered that system. Payment providers began enabling autonomous agents to initiate UPI transfers on behalf of users and businesses — without a human pressing confirm for every transaction.

India is the first country where this has happened at national scale. The infrastructure is real, the agents are live, and millions of people are already transacting through them. The problem is also real: when an AI agent initiates a payment, the receiving system can verify who sent it — but it cannot assess whether that agent should be trusted.

"Authentication tells you who an agent claims to be. Trust tells you whether to believe it."

This is the gap I set out to fix. The result is Airlock Protocol — an open trust and identity standard for autonomous AI agents. Built in India, designed for the agentic internet, submitted to the IETF as an Internet-Draft.

The Problem: AI Agents Have Identity, But Not Trust

When an agent calls an API, it authenticates with an OAuth token or an API key. That credential answers one question: Is this the agent it claims to be? Authentication is solved. What's not solved is trust.

Consider what you'd actually want to know about an agent before it moves money on your behalf:

  • Is this agent new with no track record, or established with thousands of successful interactions?
  • Has it behaved consistently within its declared capabilities, or has its behavior drifted?
  • If Agent A delegated work to Agent B, does B inherit A's trust automatically — or should that scope be narrower?
  • If Agent A is compromised, can its delegated agents still operate?
  • Is there an auditable, tamper-evident record of every verification decision?

Current infrastructure answers none of these questions. OAuth handles identity. Nothing handles trust. And as agent autonomy increases — agents booking flights, initiating transfers, signing contracts — the absence of a trust layer becomes a systemic risk.

The Scale Context

UPI processes roughly 500 million transactions every day. Even if agentic payments represent a small fraction of that today, the attack surface for compromised or unverified agents is not theoretical — it is national infrastructure.

The Analogy That Got Me Here: DMARC

Email had exactly this problem in the early 2000s. SMTP could tell you a message arrived from a server. It could not tell you whether to trust that server. Phishing, spoofing, and spam were rampant because identity and trust were conflated — and neither was enforced.

Then DMARC came along. It was an open protocol. Any domain owner could publish a trust policy. Any mail receiver could verify it. Gmail adopted it. Then Yahoo. Then everyone. Phishing dropped because the infrastructure that supported it — unauthenticated identity — was removed at the protocol level.

AI agents today are in that pre-DMARC era. Anyone can spin up an agent, give it a credential, and point it at an API. The receiving system has no standard way to check: how trustworthy has this agent been? What has it done before? Should I let it operate with full access, or restricted access, or none at all?

Airlock is the trust layer that's missing. Not a product. A protocol. Open-source. IETF-tracked. Designed to become infrastructure, the way DMARC became infrastructure.

How Airlock Works

When an agent wants to interact with a system through Airlock, it goes through five phases. The whole pipeline runs in under 100 milliseconds.

Phase 1 — Resolve

Look up the agent's profile: its declared capabilities, its registered public key, and its current trust tier. This uses W3C Decentralized Identifiers (did:key) — self-certifying identifiers that encode the agent's Ed25519 public key directly. No central authority required.

Phase 2 — Handshake with Proof of Work

The agent solves a SHA-256 computational puzzle before proceeding. This rate-limits spam and ensures an agent has spent resources to verify itself. Difficulty adapts to the agent's trust tier — well-established agents get easier puzzles because they've earned that privilege.

Phase 3 — Dual-Mode Identity Verification

This is where Airlock integrates with existing infrastructure. Agents can authenticate via:

  • Ed25519 signature — pure cryptographic proof using did:key
  • OAuth 2.1 bearer token — using existing enterprise identity providers

The OAuth path uses private_key_jwt client authentication (RFC 7523), the most secure method available — no shared secrets, no passwords. Agents that already have OAuth credentials plug into Airlock without rebuilding their identity stack.

Phase 4 — Trust Scoring and Verdict

Every agent in the Airlock registry has a trust score from 0.0 to 1.0, across four tiers:

  • Tier 0 — Unverified (0.0–0.25): New agents. Highest friction, limited access.
  • Tier 1 — Basic (0.25–0.55): Passed initial verification, demonstrated basic reliability.
  • Tier 2 — Verified (0.55–0.80): Established track record. Standard access, reduced friction.
  • Tier 3 — Trusted (0.80–1.0): Battle-tested. Full access, can delegate to other agents.

Trust scores evolve. Successful interactions increase them. Failed verifications decrease them. Temporal decay reduces them when an agent is inactive — with decay rates calibrated per tier (higher tiers decay slower). Floor protection prevents established agents from catastrophic drops due to single incidents. This is a credit score for AI agents: earned through behavior, not assumed from credentials.

Phase 5 — Seal with Auditable Attestation

The orchestrator issues a signed attestation and an OAuth 2.1 access token carrying the agent's trust claims (airlock:trust_score, airlock:trust_tier). Every verification — pass or fail — is recorded in a hash-chained audit trail. Modify any record after the fact and the chain breaks. The audit is tamper-evident.

Native Google A2A Integration

In April 2025, Google published the Agent-to-Agent (A2A) protocol — a standard for communication between autonomous AI agents across different vendors and systems. A2A defines how agents discover each other, exchange capabilities, and coordinate tasks. It is the emerging interoperability layer for the multi-agent internet.

Airlock has native A2A support built in. When one A2A agent calls another through Airlock, trust verification happens transparently — the receiving agent gets the caller's identity, trust score, and delegation chain without writing any verification code. A2A handles the communication. Airlock handles the trust.

    return initiate_upi_transfer(amount)

This is why Airlock matters specifically for the agentic payments context. When an agent on Google's A2A network initiates a UPI payment, Airlock intercepts, verifies, and either allows the transaction (with a trust-bearing token) or rejects it. The payment processor gets a cryptographically signed attestation: this agent is Tier 2, score 0.73, delegation authorized by parent agent X with scopes [payment:initiate, amount:<10000].

RFC 8693 Delegation: The Cascade Revocation Problem

Multi-agent systems have a specific trust problem that OAuth alone doesn't solve: delegation chains. Agent A initiates a task. It delegates subtasks to Agents B and C. Agent B delegates further to Agent D.

If Agent A is compromised and its token is revoked, what happens to B, C, and D? In a standard OAuth system: nothing. They keep operating. Their tokens are valid until expiry.

Airlock implements RFC 8693 Token Exchange with cascade revocation. When Agent A's trust is revoked, every token it delegated — and every token those agents delegated — is immediately invalidated. Scope narrowing ensures delegated agents can only do what they were explicitly authorized to do, not inherit the full scope of their principal.

The Compliance Layer

Regulation is coming. The EU AI Act is already in effect. India's Digital Personal Data Protection Act is in implementation. Every major framework for AI governance includes requirements for: agent accountability, audit trails, risk classification, and incident response.

Airlock's compliance module handles all of this:

  • Agent inventory — every registered agent classified by capability and risk level
  • Automated risk classification — capability-weighted and trust-score-weighted scoring
  • Incident tracking — hash-chain integrity, tamper-evident records
  • Regulatory mapping — maps verification outcomes to 7 trust principles and 10 recommendations

Organizations running agents on UPI payment rails can generate compliance reports showing every agent interaction, its verification outcome, its trust score at the time of transaction, and a complete delegation chain — in a format that survives an audit.

End-to-end verification latency averages 73ms. The cryptographic path (without network) completes in microseconds. The full five-phase pipeline — including reputation lookup and audit write — runs in under 100ms.

Why Now, Why India

India solved a problem the world thought was unsolvable: real-time payments at national scale, available to everyone, interoperable across providers, running on open standards. UPI is proof that open protocols beat proprietary networks when the problem is infrastructure.

The agentic payments problem is the same kind of problem. It requires the same kind of solution: an open protocol, not a proprietary product. A standard that any provider can implement, any agent can use, any regulator can audit.

India is where this problem is most acute because India moved fastest. The trust gap that exists globally in multi-agent systems is most visible here because agents are already operating on payment rails that process billions of transactions. Solving it here means solving it for the agentic economy that the rest of the world is building toward.

The IETF Path

The Airlock protocol specification — draft-singh-agent-trust-00 — is an IETF Internet-Draft covering the complete verification protocol, OAuth 2.1 integration, delegation semantics, and audit requirements. It passes IETF idnits validation with 0 errors. Submission to the IETF datatracker is imminent. Standards-track work, not a product announcement.
